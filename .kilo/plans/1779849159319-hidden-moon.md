# Plugin Communication Analysis: gRPC + Python Support

## Current Communication Architecture

```
┌─────────────────────────────────────────────────┐
│                 stackyrd process                 │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ TS Plugin │  │ WASM     │  │ Go Plugin     │  │
│  │ (goja VM) │  │ (wazero) │  │ (native code) │  │
│  └────┬─────┘  └────┬─────┘  └──────┬────────┘  │
│       │             │               │           │
│       │   Shared    │   Shared      │  Direct    │
│       │   JS Obj    │   Memory     │  Go call   │
│       ▼             ▼               ▼           │
│  ┌──────────────────────────────────────────┐   │
│  │           PluginBridge                   │   │
│  │  (same process, same address space)      │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**Key characteristics:**
- All plugins are **in-process** — zero serialization overhead
- WASM uses shared linear memory (pointer+length ABI)
- TS uses direct JS object references in goja VM
- Go plugins are direct function calls
- Communication is synchronous, blocking, and cheap

## Analysis: gRPC for Plugin Communication

### Would it be an improvement? **It depends on the goal.**

#### Case FOR gRPC (net positive):

| Factor | Current | gRPC |
|--------|---------|------|
| **Cross-language** | Go, TS, WASM-only | Any language with gRPC (Python, Rust, Java, C#, etc.) |
| **Out-of-process** | Impossible | Plugins can run on other hosts |
| **Contract enforcement** | Implicit (JSON shape) | Explicit (protobuf schema) |
| **Streaming** | Not supported | Bidirectional streaming |
| **Versioning** | Manual migration | Protobuf wire compat |
| **Auth/TLS** | N/A | Built-in mTLS support |

#### Case AGAINST gRPC (net negative for current use):

| Factor | Current | gRPC |
|--------|---------|------|
| **Latency** | ~1-10μs per call (shared mem) | ~1-10ms per call (serialize+network) |
| **Complexity** | ~20 lines of host ABI code | Proto files, codegen, server, client stubs, connection pool |
| **Resource usage** | Runs inside existing process | Separate process per plugin (RAM, CPU, ports) |
| **WASM support** | Natural (linear memory ABI) | Impossible (WASM can't run gRPC server) |
| **TS support** | Natural (goja in-process) | Impossible (goja can't run gRPC) |
| **Deployment** | Single binary | Plugin processes + orchestration |
| **Error modes** | Panic → recover | Network failures, timeouts, reconnection |
| **Debugging** | Single process, simple | Distributed tracing needed |

### Recommendation: **Hybrid — keep in-process, add gRPC as optional extension**

Don't replace the current in-process communication. Instead, introduce a **`GRPCPlugin`** type that implements the same `Plugin` interface but communicates over gRPC. This way:

- **TS plugins** stay in-process via goja (fast, cheap)
- **WASM plugins** stay in-process via wazero (fast, sandboxed)
- **Go plugins** stay as direct calls (fastest)
- **Python/other-language plugins** use gRPC as an external runtime

The `PluginBridge` API (`bridge.go`) doesn't change — it already abstracts over the `Plugin` interface. A new `GRPCPlugin` just implements `Execute()` by making a gRPC call instead of running code in-process.

## Python Support Strategy

### Option 1: gRPC-based Python Plugin Runtime (Recommended)

```
┌─────────────────────┐     gRPC     ┌──────────────────────┐
│   stackyrd          │◄───────────►│  python-plugin-host   │
│                     │             │  (separate process)    │
│  PluginBridge ──────┼─stream─────►│                       │
│       │             │             │  ┌──────────────────┐  │
│  ┌────┴──────┐      │             │  │  Python Plugin   │  │
│  │ GRPCPlugin│      │             │  │  (any framework) │  │
│  │ (client)  │      │             │  └──────────────────┘  │
│  └───────────┘      │             └──────────────────────┘
└─────────────────────┘
```

**How it works:**
1. A Python process (`python-plugin-host`) runs as a sidecar, started either by stackyrd or independently
2. It registers itself with the gRPC server embedded in stackyrd (or connects as a client)
3. `GRPCPlugin` wraps a gRPC client stub implementing the `Plugin` interface
4. Python plugins are defined as Python files/modules loaded by the host process

**Proto contract:**
```protobuf
service PluginRuntime {
  rpc Execute(ExecuteRequest) returns (ExecuteResponse);
  rpc Ping(Empty) returns (Pong);
}

message ExecuteRequest {
  string plugin_name = 1;
  bytes args_json = 2;
  string script_source = 3;  // Loaded from store overlay
}

message ExecuteResponse {
  bool success = 1;
  bytes data_json = 2;
  string error = 3;
}
```

**Pros:** Clean separation, Python ecosystem fully available, natural for gRPC
**Cons:** Process management overhead, gRPC latency

### Option 2: Embedded Python via go-python3

Directly embed CPython into the Go process using `go-python3`:
- Python scripts run in the same process
- No serialization overhead
- **But:** CPython GIL issues, version lock-in, CGo overhead, build complexity, segfault risk

**Not recommended** for production use due to CGo fragility and version coupling.

### Option 3: Subprocess with JSON-RPC

Python runs as a subprocess, communicates via stdin/stdout with JSON-RPC:
- Simpler than gRPC (no protobuf, no codegen)
- No port management
- **But:** No streaming, manual framing, less robust error handling, no auth/TLS

**Viable for simple cases** but gRPC is more robust long-term.

## Recommended Implementation Plan

### Phase 1: gRPC Infrastructure

| Step | File(s) | Description |
|------|---------|-------------|
| 1 | `pkg/plugin/plugin.proto` | Define `PluginRuntime` gRPC service |
| 2 | Generate Go stubs | `protoc --go_out=. --go-grpc_out=.` |
| 3 | `pkg/plugin/grpc_runtime.go` | `grpcRuntime` implementing `Runtime` interface for `"grpc:"` prefix |
| 4 | `pkg/plugin/grpc_plugin.go` | `GRPCPlugin` — `Plugin` impl that dials a gRPC endpoint, calls `Execute` |
| 5 | `pkg/plugin/grpc_server.go` | Optional: embed gRPC server in stackyrd for Python sidecar to connect to |
| 6 | Wire into `init.go` | Register `grpcRuntime` via `init()`, handle `"grpc:"` entrypoints |

### Phase 2: Python Plugin Host

| Step | File(s) | Description |
|------|---------|-------------|
| 1 | `scripts/plugins/python/host.py` | Python gRPC server that loads and runs Python plugins |
| 2 | `scripts/plugins/python/plugin.proto` | Same proto, Python stubs via `grpc_tools` |
| 3 | `pkg/plugin/builtin/*/plugin.yaml` | Example Python plugin with `entrypoint: "grpc:scripts/handler.py"` |
| 4 | Plugin SDK for Python | `scripts/plugins/python/sdk.py` — helper class for writing plugins |
| 5 | Docs | Update PLUGIN_PKG.md + AGENTS.md |

### Phase 3: Integration

| Step | Description |
|------|-------------|
| 1 | `docker-compose.yaml` — add Python plugin host service |
| 2 | Health check / reconnection logic in `GRPCPlugin` |
| 3 | Config — `plugins.grpc` section for endpoints, timeouts, TLS |
| 4 | `PluginBridge.Execute` — transparently handles in-process vs gRPC plugins |

## Tradeoffs Summary

| Aspect | Recommendation |
|--------|---------------|
| Replace current plugin ABI? | **No** — keep shared-memory for WASM/TS |
| Add gRPC as a new runtime? | **Yes** — for external/Python plugins |
| Python support strategy | **gRPC sidecar** — not embedded Python |
| Proto contract scope | Keep it narrow: `Execute`, `Ping`, optional `StreamLogs` |
| Who starts the Python host? | Configurable: stackyrd can exec it, or run independently |

---

## Questions for the user

1. Do you want to add gRPC as an **alternative** runtime (keeping TS/WASM/Go in-process), or as a **replacement** for all plugins?
2. For Python: would plugins be Python scripts loaded from `builtin/` (like TS plugins), or would they be standalone Python services with their own lifecycle?
3. Should the gRPC server run inside stackyrd (Python connects to it), or should Python run its own gRPC server that stackyrd connects to?
